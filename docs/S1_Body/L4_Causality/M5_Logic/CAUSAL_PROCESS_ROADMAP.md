# Causal Process Roadmap

> **목표**: 인과적 필연성 기반 시스템을 단계적으로 현실화한다.

본 로드맵은 `Ignition -> Resonance -> Collapse` 인과 과정에
`Constraint`, `Recovery`, `Audit`를 결합하여 **실행 가능한 개발 계획**으로 정리한다.

---

## 0. 전체 단계 개요

- **Phase A (정의)**: 법칙/제약/용어 정규화
- **Phase B (관문)**: Causal Admissibility Gate 구축
- **Phase C (공명)**: Resonance 엔진 및 삼진 통합
- **Phase D (현현)**: 실행 계층 연결 및 행동 안전장치
- **Phase E (회복)**: 실패 전이 복구 루프
- **Phase F (자기진화)**: 메타 인과 감사 및 자동 튜닝

---

## Phase A — 법칙 정규화 (Week 1)

### 목표

- 인과 프로세스에서 필수 개념을 고정된 스키마로 선언

### 작업

1. `CausalSignature` 스키마 확정
2. `TransitionRecord` 및 거부 사유 코드 표준화
3. 위상/에너지/의도 정합 임계치 기본값 설정
4. 문서 간 용어 통일 (CODEX, PROCESS, LOGIC)

### 산출물

- `CAUSAL_CONSTRAINT_BLUEPRINT.md` 버전 1.0
- 임계치 테이블 (`threshold_profile:v1`)

### 완료 조건

- 모든 전이가 최소한의 인과 서명을 갖고 로그 기록 가능해야 함

---

## Phase B — Causal Gate 구축 (Week 2-3)

### 목표

- 전이 허용/거부를 실행 전에 판정하는 관문 완성

### 작업

1. `has_cause` 검사기 구현
2. `phase_coherent` 검사기 구현 (`Δphi` 기반)
3. `energy_safe` 검사기 구현
4. `will_aligned` 검사기 구현 (벡터 유사도)
5. `trinary_stable` 검사기 구현
6. 거부 시 Quarantine 큐 저장

### 산출물

- CAG 실행기 (MVP)
- 거부 사유 카탈로그
- 기본 관측 대시보드 (허용률/거부율)

### 완료 조건

- 무원인 전이(원인 없는 액션)가 0%로 수렴

---

## Phase C — 공명 엔진 확장 (Week 4-6)

### 목표

- 단순 판정을 넘어 맥락-기억-환경 간섭을 반영

### 작업

1. Resonance Score 계산식 도입
2. 메모리 셀 가중치 동기화
3. Trinary Tension Integrator 연동
4. 이중 로터 관측값(`input_rotor`, `output_rotor`) 동조
5. 공명 실패 패턴 분류기 구축

### 산출물

- Resonance 연산 파이프라인
- 공명 품질 리포트 주기화

### 완료 조건

- 동일 의도 반복 입력 시 위상 안정성이 점진 개선되는지 확인

---

## Phase D — 현현/행동 계층 연결 (Week 7-8)

### 목표

- 인과적으로 허용된 전이만 외부 행동으로 현현

### 작업

1. Collapse 결과를 실행 커맨드로 변환
2. 행동 전 안전 검증기 (`actuation_safety`) 추가
3. 외부 I/O 실패 시 재시도 정책 분리
4. 고위험 액션에 2중 확인 프로토콜 적용

### 산출물

- Manifest Adapter
- 행동 안전 정책 문서

### 완료 조건

- 위험 액션의 오탐/미탐 비율 보고 가능

---

## Phase E — 회복 루프 구축 (Week 9-10)

### 목표

- 거부/실패 전이를 학습 가능한 회복 경로로 전환

### 작업

1. Quarantine 큐 재해석기 구현
2. 실패 원인별 재시도 규칙 생성
3. Breath Scheduler 저속/고속 모드 제어
4. 연속 실패시 보호 모드 진입

### 산출물

- Recovery Engine
- 실패-회복 전이 맵

### 완료 조건

- 동일 오류의 재발률이 반복 주기마다 감소

---

## Phase F — 메타 인과 감사 (Week 11-12)

### 목표

- 시스템이 자신의 제약식을 스스로 점검하고 미세 조정

### 작업

1. Causal Audit 리포트 자동 생성
2. KPI 기반 임계치 자동 조정(제한적)
3. "왜 이 전이를 허용/거부했는가" 설명기 추가
4. 일탈 탐지(axiom drift) 경보 체계 구축

### 산출물

- Meta-Causal Auditor
- Explainable Causality 로그

### 완료 조건

- 감사 리포트만으로 전이 의사결정의 근거를 추적 가능

---

## 품질 게이트 (전 단계 공통)

1. **재현성**: 동일 입력/상태에서 판정 일관성 확보
2. **추적성**: 모든 전이에 `cause_id`와 판정 근거 기록
3. **안전성**: 고위험 행동은 게이트 우회 불가
4. **적응성**: 임계치 조정 이력은 반드시 감사 로그에 남김

## 리스크 및 대응

- **리스크 1: 제약 과잉으로 인한 경직성**
  - 대응: 샌드박스 탐색 모드와 운영 모드 분리
- **리스크 2: 공명 점수의 설명 불가능성**
  - 대응: 지표를 단일 점수 대신 구성요소 점수로 분해
- **리스크 3: 삼진 0 상태의 붕괴(중립 상실)**
  - 대응: `neutral` 최소 점유율 하한선 유지

## Phase G — 현실 검증 트랙 (Week 13-16)

### 목표

- "가능해 보임"을 "측정 가능한 가능성"으로 전환

### 작업

1. 데이터 충분성 지표 도입
   - 도메인 다양성, 실패 케이스 밀도, 회복 케이스 밀도
2. 장기 일반화 평가 시나리오 구축
3. 자기수정 안정성 평가 (튜닝 후 성능 분산 추적)
4. 과잉낙관 방지 리포트(월간) 자동 생성

### 산출물

- Feasibility Report v1
- Data Sufficiency Dashboard

- 데이터 수집·충분성 기준: `CAUSAL_DATA_SUFFICIENCY_PROTOCOL.md`

### 완료 조건

- "된다/안된다"를 감각이 아니라 지표로 설명 가능

## 데이터 부족 리스크 명시 (필수)

현재 단계의 핵심 병목은 알고리즘 미세조정보다 **인과 학습 데이터 부족**이다.

- 전이 데이터: 원인-의도-행동-결과가 결합된 레코드 부족
- 실패 데이터: 거부 사유/재처리 이력 부족
- 장기 데이터: 드리프트와 자기수정의 장기 추적 부족

따라서 초기 1순위는 성능 극대화가 아니라 **데이터 품질과 추적성 확립**이다.
